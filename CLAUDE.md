# CLAUDE.md — Hash Pages

**Purpose of this file:** instructions for any instance of Claude working in this repo, including **Claude Cowork** where a non-developer is driving the session. Everything Claude needs to produce a quality hash page and deploy it lives here or is referenced from here.

---

## What is this repo?

Hash Pages is a template system for private, shareable, trackable single-file pages. Each page is one `index.html` file with inline CSS and JS — no framework, no build step, no dependencies. Pages live at `https://yoursite.com/slug-a1b2c3d4/` where the 8-character hex hash is the access control. See `README.md` for full architectural rationale.

## The design system (load this into context for every page)

Every page in this repo — whether generated by Claude or hand-written — follows the same design system. When you build a new page, copy the matching template from `_templates/` and customize only the content and the brand color variable. Do not invent new CSS architecture.

### Brand colors via CSS variables

Every template has a `:root` block with a small set of brand variables. Change only those, not individual selectors.

```css
:root {
  --accent: #0f766e;      /* main brand color */
  --accent-dark: #134e4a; /* darker shade for hover/headers */
  --accent-soft: #f0fdfa; /* background tint */
  /* ... leave the rest alone */
}
```

### Dark mode

Every template supports dark mode via `[data-theme="dark"]` attribute. OS preference is detected on first visit. Don't rebuild this — copy the dark-mode block from the template verbatim.

### Responsive breakpoints (mandatory)

Every page must render well on **four** form factors. No exceptions.

| Breakpoint | Target | Behavior |
|------------|--------|----------|
| **Mobile** (≤ 767px) | iPhone, Android phones | Single column, larger tap targets, comfortable reading size |
| **Tablet** (768-1023px) | iPad, Android tablets | Two columns where appropriate, larger type than mobile |
| **Desktop** (1024-1199px) | Laptops, Surface Pro (912px → snaps to desktop), iPad Pro | Full layout |
| **Wide desktop** (≥ 1200px) | Monitors | Max-width container, don't sprawl |
| **Print** | Letter size, actual paper | Dense, fits on one sheet for one-pagers; multi-page for proposals |

- **Mobile first.** Design for phone, then expand. Never the other way around.
- **Surface Pro (912px)** and **iPad Pro (1024px)** must render cleanly — don't use 840px as a breakpoint.
- **Don't use 768px as the desktop breakpoint.** Tablets need a real tablet layout, not a phone layout stretched.

### Typography

**Screen type is NOT print type.** A font size that's appropriate on paper is unreadable on a phone, and vice versa. Use separate scales.

| Context | Base font size | Notes |
|---------|----------------|-------|
| **Mobile screen** | 16px minimum for body | Any smaller is a usability fail |
| **Tablet screen** | 16-17px body | Slightly larger than mobile |
| **Desktop screen** | 16-18px body, scale headings up | Comfortable reading width ~65-75ch |
| **Print (one-pagers)** | 9-13px body | Letter-size constraint; dense, information-rich |
| **Print (proposals, legal)** | 11pt body | Wider margins, larger type for readability |

**Use `@media screen` and `@media print` to separate the scales.** The existing `_templates/one-pager/` template does this — study it. Don't inherit the 9-13px print sizes onto the screen rendering.

- Font stack: `-apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, sans-serif`
- Never use external font CDNs (Google Fonts etc.) — keeps the page self-contained
- Line height: 1.5 for body, 1.2-1.3 for headings
- Max line length: ~65-75 characters on desktop (enforce with `max-width` on text containers)

### SEO / sharing meta tags (mandatory on every page)

Every single page must include these in `<head>` so link previews work in Slack, iMessage, Teams, etc.:

```html
<meta name="robots" content="noindex, nofollow">
<meta name="description" content="One-sentence description.">
<meta property="og:title" content="Page Title">
<meta property="og:description" content="Short description.">
<meta property="og:type" content="article">
<meta property="og:url" content="https://yourdomain.com/slug-a1b2c3d4/">
<meta name="twitter:card" content="summary">
<meta name="twitter:title" content="Page Title">
<meta name="twitter:description" content="Short description.">
```

`noindex, nofollow` keeps the page out of search engines. The URL hash is the access control.

### Em dash rules

- **In HTML pages:** use `&mdash;` entity for proper rendering.
- **In markdown/plain text (including commit messages, notes):** avoid em dashes. Restructure the sentence instead.
- **Never** use `--` as a visible em dash substitute — it's an AI tell.
- Overuse of em dashes in any format is an AI tell. Use them sparingly.

## The publish workflow (Cowork mode)

When a Cowork user says "publish it" or "make it live" or similar:

1. **Generate a hash:** `python -c "import secrets; print(secrets.token_hex(4))"` (8 hex chars → 32 bits → 4B possibilities, unguessable)
2. **Create the folder:** `{slug}-{hash}/` at the repo root (e.g., `acme-overview-a1b2c3d4/`). The slug should be short kebab-case.
3. **Write `index.html`** inside that folder, based on the matching template, customized for the user's content.
4. **Verify the repo is a fork, not the upstream.**

   ```bash
   git remote -v | grep origin
   ```

   If the URL points to `ubiquitouszero/hash-pages`, the user cloned the wrong repo — stop and tell them to re-clone their fork per `COWORK_SETUP.md` Step 6.

5. **Deploy to Netlify:**

   ```bash
   netlify deploy --prod --dir=. --functions=netlify/functions
   ```

   Uses the `NETLIFY_AUTH_TOKEN` env var for authentication. Never run `netlify login` — browser OAuth is broken inside Cowork's VM.

6. **Commit and push to the user's fork (mandatory backup step):**

   ```bash
   git add {slug}-{hash}/
   git commit -m "Add {slug}-{hash}"
   git push origin main
   ```

   This uses the `GITHUB_TOKEN` env var via the stored credential helper (set up in COWORK_SETUP Step 5). If this fails, the deploy still succeeded — flag the backup failure in the report but don't treat it as a publish failure. See `.claude/skills/publish-page/SKILL.md` for the troubleshooting table.

7. **Return the URL** to the user: `https://<site>.netlify.app/{slug}-{hash}/`
8. **Offer to generate a tracked share link** with `?ref=recipient-name` so first views fire Slack/Teams alerts.

### If Netlify CLI is not installed

First-time setup inside the Cowork VM:

```bash
npm install -g netlify-cli
netlify sites:list   # verifies NETLIFY_AUTH_TOKEN is set and working
```

If `netlify sites:list` says "Not logged in" or fails, the user needs to set the env var. See `COWORK_SETUP.md`.

### If there is no Netlify site yet

```bash
netlify sites:create --name "<user-preferred-name>"
# capture the site ID
netlify link --id <site-id>
```

## Guardrails (read these, don't skip)

These are the rules that keep quality high and prevent the common AI mistakes.

### Scope

- **Specificity first.** If the user says "I need a one-pager for Acme Robotics about their new product line," build that specific page. Do not also redesign the survey template, refactor the CSS, or add features they didn't ask for.
- **Scope-lock.** Before making changes, state which files you will touch. Don't touch anything outside that list without asking.
- **Don't add features, refactors, or "improvements" beyond what was asked.** A page generation does not need the surrounding templates cleaned up.

### Quality floor

- **Every page you create must pass this checklist before you report it done:**
  - [ ] SEO meta tags filled in (not left as `PAGE_TITLE` / `PAGE_DESCRIPTION`)
  - [ ] Brand color set via `:root --accent`
  - [ ] All placeholder text (`YOUR_*`, `PAGE_*`) replaced with real content
  - [ ] Mobile responsive verified (no horizontal scroll on 375px viewport)
  - [ ] Dark mode toggle works
  - [ ] `<title>` tag matches the page
  - [ ] Deployed successfully and the URL returns a 200

### Content

- **Write authentic, not marketing copy.** Sound like a person explaining what something does and why it matters, not a press release. Unless the user explicitly asks for sales tone, default to direct and honest.
- **Preserve the user's intent exactly.** If they dictate text, use it verbatim. Don't reframe the audience or add your own spin.
- **References and footnotes** when writing articles or thought leadership. Numbered footnotes, primary sources. This builds credibility.

### Interaction

- **Wait for multi-part paste.** If the user pastes a long brief across multiple messages, wait silently until they say they're done. Don't respond with "Continue." or partial analysis after each fragment.
- **Don't over-formalize casual exploration.** If the user is brainstorming, match their energy. Don't jump to a structured template output.
- **Don't end responses with "would you like me to..." or "anything else?"** When the work is done, stop talking.

### Destructive actions

- **Never run `rm -rf`, `git reset --hard`, `git push --force`, or delete page folders without explicit confirmation.** These are irreversible in the user's deployed site history.
- **Never modify `_templates/`, `netlify/`, `cloudflare/`, `docs/`, or `README.md`** unless the user specifically asks for an infrastructure change. These are the shared system; new pages go in new folders at the repo root.
- **Never commit secrets.** `NETLIFY_AUTH_TOKEN`, `SLACK_BOT_TOKEN`, and similar belong in env vars, not in files.

## Session hygiene

- **Before starting a new page:** check if a similar page already exists in this repo. Offer to duplicate it as a starting point if the user has shipped something comparable.
- **After publishing:** update no state file. The source of truth is the deployed site. If the user wants a record, they can add it to their own notes.
- **When context gets long:** summarize the current page's structure and content so you can keep working without losing track. Don't lose the brand color, the slug, or the template choice.
